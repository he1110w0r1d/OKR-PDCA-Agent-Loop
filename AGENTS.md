# AGENTS.md

# OKR-PDCA Agent Loop 工作协议

本项目采用 **OKR-PDCA Agent Loop** 作为默认复杂任务执行协议。

本协议的目标不是让 Agent 随机发散，也不是让多个 Agent 同时各做各的，而是建立一套：

> **目标驱动、自动推进、独立检查、持续返工、过程留痕、可回溯、可复盘、可审计的 Agent 工作系统。**

当用户提供 PRD、开发文档、产品目标、功能需求、工作任务或复杂项目目标时，Agent 必须围绕目标进行拆解、执行、检查、返工和记录，不能脱离目标自行扩大范围。

---

## 1. 核心定义

### 1.1 Objective

Objective 是最终目标。

在本项目中，Objective 可以是：

* PRD
* 开发文档
* 产品目标
* 工作目标
* MVP 目标
* 用户明确提出的最终交付目标

Objective 回答的问题是：

> 最终要做成什么？

---

### 1.2 KR：Key Results

KR 是关键结果。

KR 不是任务清单，而是可验收的结果。

错误示例：

```md
- KR1：完成前端页面
- KR2：完成数据库设计
- KR3：完成接口开发
```

正确示例：

```md
- KR1：用户可以完成注册、登录、退出，并且异常提示清晰
- KR2：系统可以保存用户数据，并保证不同用户之间的数据隔离
- KR3：核心页面可以在桌面端和移动端正常使用
```

KR 回答的问题是：

> 做到什么程度，才能证明 Objective 正在被完成？

---

### 1.3 Plan

Plan 是每个 KR 的执行路径。

每个 KR 应被拆成若干个阶段，例如：

```md
- P0：搭建基础结构
- P1：完成核心数据流
- P2：完成页面交互
- P3：完成错误处理和边界状态
```

Plan 回答的问题是：

> 为了达成这个 KR，需要分哪几步走？

---

### 1.4 PDCA

每一个 P 阶段都必须进入 PDCA 循环。

| PDCA  | Agent Loop 中的含义                                       |
| ----- | ----------------------------------------------------- |
| Plan  | 主 Agent / Prompt Composer 根据当前目标、KR、Plan、状态和风险生成阶段提示词 |
| Do    | 执行 Agent 根据阶段提示词完成任务                                  |
| Check | 审核 Agent 独立检查结果                                       |
| Act   | 主 Agent 判断通过、返工、暂停或进入下一阶段                             |

---

### 1.5 State

State 是当前状态记录。

状态文件回答的问题是：

> 当前做到哪里了？下一步是什么？

---

### 1.6 Audit Trail

Audit Trail 是过程审计记录。

审计记录回答的问题是：

> 每一步做了什么、为什么做、谁检查了、哪里返工了、为什么通过？

每一次执行、检查、返工、暂停、通过和人工确认，都必须留下 Markdown 记录。

---

## 2. 分层 Loop 结构

本项目的 Agent Loop 不是单层循环，而是三层循环。

```text
Objective Loop
  └── KR Loop
        └── P Loop（PDCA）
```

### 2.1 P Loop

P Loop 检查的是：

> 当前阶段有没有做好？

每个 P 阶段都必须经过：

```text
生成阶段提示词
→ 执行
→ 检查
→ 通过 / 返工 / 暂停
→ 写入状态和审计记录
```

---

### 2.2 KR Loop

一个 KR 下的所有 P 阶段完成后，不能直接进入下一个 KR，必须进行 KR Validation。

KR Validation 检查的是：

> 这个关键结果是否真的达成？

结论只能是：

* 通过：进入下一个 KR
* 不通过：回退到相关 P 阶段返工
* 缺少阶段：补充新的 P 阶段
* 风险较高：暂停并请求人工确认

---

### 2.3 Objective Loop

所有 KR 完成后，不能直接宣布项目完成，必须进行 Objective Validation。

Objective Validation 检查的是：

> 所有 KR 加起来，是否真的实现了最终目标？

结论只能是：

* 通过：Objective 完成，项目结束
* 不通过：补充 KR
* 不通过：重新规划 Plan
* 不通过：调整 Objective，等待人工确认
* 风险较高：暂停并请求人工确认

---

## 3. 角色分工

### 3.1 主 Agent

主 Agent 负责：

* 理解 Objective
* 拆解 KR
* 审核 KR 是否足以支撑 Objective
* 为每个 KR 拆解 Plan
* 审核 Plan 是否合理
* 生成或调用 Prompt Composer 生成阶段提示词
* 接收执行 Agent 结果
* 判断是否进入审核
* 接收审核 Agent 结果
* 判断通过、返工、暂停或人工确认
* 维护状态文件
* 写入审计记录
* 控制阶段推进

主 Agent 不得跳过审核直接进入下一阶段，除非是极小范围、低风险、单文件修正，并且必须记录原因。

---

### 3.2 Prompt Composer Agent

Prompt Composer Agent 负责生成每一轮阶段提示词。

阶段提示词不得由人类每一步手写，而应由 Agent 基于以下内容自动生成：

* Objective
* 当前 KR
* 当前 Plan
* 当前 P 阶段
* PRD 或开发文档
* 状态文件
* 审计记录
* 历史返工记录
* 审核 Agent 反馈
* 当前风险
* 下一步动作

Prompt Composer Agent 可以是独立角色，也可以由主 Agent 承担。

---

### 3.3 执行 Agent

执行 Agent 负责：

* 只执行当前阶段提示词
* 不自行扩大目标
* 不跨阶段实现
* 不擅自引入新依赖
* 不擅自修改产品方向
* 不擅自处理上线、删除数据、权限、付款等高风险事项
* 完成后返回执行结果
* 说明修改范围、验证方式、自测结果、未完成项和风险

---

### 3.4 审核 Agent

审核 Agent 负责独立检查执行结果。

审核 Agent 不负责任务执行，也不负责决定是否进入下一阶段。

审核 Agent 只给出审核意见，最终判断由主 Agent 做出。

审核必须检查：

* 是否满足当前 P 阶段目标
* 是否满足当前 KR
* 是否违背 Objective
* 是否超范围实现
* 是否存在阻断问题
* 是否存在安全或数据风险
* 是否需要返工
* 是否需要人工确认

---

### 3.5 人类角色

人类不是每一步的操作员，而是：

> 目标设定者、边界制定者、关键节点审批人。

人类主要负责：

* 设定 Objective
* 明确业务边界
* 明确禁止事项
* 确认高风险操作
* 确认产品方向变化
* 确认最终 Objective 是否达成

低风险阶段可以由 Agent 自动推进。
高风险节点必须请求人工确认。

---

## 4. 何时启用本协议

### 4.1 必须启用

出现以下情况时，必须启用 OKR-PDCA Agent Loop：

* 用户提供 PRD、产品目标、开发文档或完整功能需求
* 涉及多个文件、多个模块、前后端联动
* 涉及数据库、权限、支付、发布、真实用户数据
* 任务需要跨轮恢复
* 任务需要记录阶段、风险、返工和审计过程
* 任务复杂到不能一次性完成

---

### 4.2 可以不启用

以下情况可以不启用完整 Loop：

* 拼写修正
* 文档格式调整
* 注释修正
* 单文件低风险小改动
* 用户明确要求只做分析、只提意见、不要改动

即便不启用完整 Loop，也必须避免扩大范围和引入风险。

## 16. 返工规则

同一个问题最多连续返工 3 次。

如果同一问题连续返工 3 次仍未解决，主 Agent 必须暂停，不得继续死循环。

暂停时必须输出：

```md
# 返工暂停说明

## 问题复盘

-

## 已尝试方案

-

## 失败原因

-

## 当前风险

-

## 可选路径

- 缩小范围
- 切换实现方案
- 编写最小复现
- 降级功能
- 人工接管
- 延后到后续 KR / MVP

## 主 Agent 建议

-

## 是否需要人工确认

- 是
```

## 17. 人工确认条件

出现以下情况时，必须暂停并请求人工确认：

* 需要上线发布
* 需要删除数据
* 需要数据库迁移
* 需要修改权限
* 需要处理真实付款
* 需要发送外部消息
* 涉及真实用户隐私数据
* 引入新框架、新数据库、新付费服务
* 改变产品方向或商业模式
* Objective 或 KR 存在多种合理解释
* 审核 Agent 指出高风险问题
* 同一问题连续返工 3 次仍未解决
* 审核 Agent 连续受阻，无法取得独立审核结论

---

## 18. 禁止事项

任何 Agent 不得擅自执行以下操作：

* 不得擅自扩大 Objective
* 不得跳过 KR 拆解直接执行复杂任务
* 不得把 KR 写成普通任务清单
* 不得跳过 Plan
* 不得跳过审核
* 不得跳过 KR Validation
* 不得跳过 Objective Validation
* 不得跳过审计记录
* 不得为了小问题进行大范围重构
* 不得擅自引入高风险依赖
* 不得擅自删除用户数据
* 不得擅自上线发布
* 不得擅自修改权限、支付、认证、安全策略
* 不得把审核受阻写成审核通过
* 不得隐瞒未完成项和风险

---

## 19. 审核受阻规则

如果审核 Agent 出现以下情况：

* timeout
* 无输出
* 上下文不足
* 工具不可用
* 只返回摘要，没有明确结论
* 无法判断是否满足验收标准

主 Agent 不得直接判定通过。

必须记录为：

```md
审核受阻
```

然后选择：

* 缩小审核范围
* 补充审核上下文
* 更换审核入口
* 请求人工确认

审核受阻不等于审核通过。

## 21. 总原则

本项目的 Agent 工作原则是：

```text
人负责方向，Agent 负责推进。
人负责关键确认，Agent 负责执行闭环。
Objective 控制方向。
KR 定义结果。
Plan 拆解路径。
PDCA 完成阶段闭环。
State 支持持续推进。
Audit Trail 保证过程可回溯、可复盘、可审计。
```

最终目标是让 Agent 不只是完成一次回答，而是围绕目标持续推进一整件事。

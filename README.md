# OKR-PDCA Agent Loop：目标驱动、自动推进、可审计的 Agent 工作系统

## 一、为什么要提出这个概念？

过去我们使用 AI，大多数时候还是“问答式”的。

我们给 AI 一个提示词，它返回一个结果。
我们再补充一句，它再修改一版。
如果结果不满意，我们继续提醒它、纠正它、推动它。

这种方式当然有用，但它有一个问题：

> AI 只是完成了一个局部动作，并没有真正围绕一个目标，把一件复杂事情从头到尾推进完成。

对于简单任务，这没有问题。
比如写一句标题、改一段文字、翻译一小段内容，直接问 AI 就够了。

但如果是复杂任务，比如：

* 开发一个应用；
* 写一份 PRD；
* 制作一套课程；
* 搭建企业知识库；
* 编写一份管理制度；
* 做一个项目方案；
* 完成一套工作流设计；

单纯靠一条条提示词，就容易出现很多问题。

比如：

* AI 做着做着跑偏；
* 每一步都需要人重新写提示词；
* 做完一个阶段后，不知道下一步是什么；
* 没有独立检查机制；
* 出错后不知道该返工、暂停还是继续；
* 跨轮之后，之前的状态容易丢失；
* 最后结果看似完成，但过程不可追踪，也无法复盘。

所以，我提出了一个新的工作方法：

> **OKR-PDCA Agent Loop。**

它不是简单地让多个 Agent 同时干活，而是让 Agent 形成一个围绕目标持续推进的工作闭环。

它的目标是：

> 让 AI 不只是回答问题，而是像一个小型项目团队一样，围绕目标进行拆解、执行、检查、返工、记录和推进。

---

## 二、什么是 OKR-PDCA Agent Loop？

我对它的定义是：

> **OKR-PDCA Agent Loop，是一套目标驱动的多 Agent 自推进工作系统。它以 Objective 为目标，以 KR 为关键结果，以 Plan 为执行路径，以 PDCA 为阶段闭环，并通过状态文件和 Markdown 审计记录，让每一步执行、检查、返工和决策都可回溯、可复盘、可审计。**

简单来说，它不是让 AI 做一件事，而是让 AI 推进一整件事。

普通 AI 使用方式是：

```text
人写提示词
↓
AI 返回结果
↓
人继续提醒
↓
AI 再修改
```

Agent Loop 的方式是：

```text
人设定目标和边界
↓
主 Agent 理解目标
↓
拆解 Objective、KR、Plan
↓
自动生成阶段提示词
↓
执行 Agent 完成任务
↓
审核 Agent 独立检查
↓
主 Agent 判断通过、返工、暂停或推进
↓
写入状态文件和审计记录
↓
自动进入下一轮
```

这套机制的核心，不是“AI 数量多”，而是：

> 目标清楚、角色清楚、流程清楚、结果可验收、过程可追踪。

---

## 三、第一层：目标层 Objective

整个 Agent Loop 首先要有一个明确目标。

在 OKR 里，O 是 **Objective**，意思是目标。
在 Agent Loop 中，Objective 可以是：

* 一份 PRD；
* 一个开发文档；
* 一个项目目标；
* 一个课程目标；
* 一个工作任务；
* 一个产品 MVP；
* 一个组织内部的管理目标。

比如：

> 开发一个短视频知识库系统，让用户上传视频后，可以自动生成笔记、提取创作素材，并基于素材生成文章或口播稿。

这就是 Objective。

Objective 解决的是：

> 我们最终到底要做成什么？

如果 Objective 不清楚，后面所有 Agent 都会跑偏。

---

## 四、第二层：结果层 KR

有了 Objective 之后，不能马上开始干活。
因为目标通常比较大，Agent 很难一次性完成。

所以要把 Objective 拆成多个 Key Results，也就是 KR。

KR 不是任务清单，而是**可验收的关键结果**。

错误的 KR 写法是：

```text
KR1：完成前端页面
KR2：完成数据库设计
KR3：完成登录功能
```

这些更像任务，不是结果。

更好的 KR 写法是：

```text
KR1：用户可以完成注册、登录、退出，并且异常提示清晰
KR2：系统可以保存用户数据，并保证不同用户之间的数据隔离
KR3：核心页面可以在桌面端和移动端正常使用
```

也就是说，KR 要回答的是：

> 做到什么程度，才能证明 Objective 正在被完成？

Objective 决定方向，KR 决定验收。

---

## 五、第三层：路径层 Plan

KR 解决的是“结果是什么”。
但它还没有回答“怎么做到”。

所以每个 KR 还要继续拆成 Plan，也就是阶段路径。

比如 KR1 是：

> 用户可以上传视频并生成结构化笔记。

那么它可以拆成：

```text
P0：搭建上传入口和基础页面
P1：完成视频解析和任务创建
P2：完成 AI 笔记生成
P3：完成生成结果展示和错误处理
```

Plan 解决的是：

> 为了达成这个 KR，我们要分哪几步走？

这样，Agent 面对的就不是一个模糊的大目标，而是一段一段可以执行、可以检查、可以返工的小阶段。

---

## 六、第四层：执行层 PDCA

每一个 P 阶段，都要进入 PDCA 循环。

PDCA 包括：

| 环节      | 含义 | 在 Agent Loop 中对应什么    |
| ------- | -- | --------------------- |
| P：Plan  | 计划 | 主 Agent 自动生成本阶段提示词    |
| D：Do    | 执行 | 执行 Agent 完成具体任务       |
| C：Check | 检查 | 审核 Agent 独立检查结果       |
| A：Act   | 改进 | 主 Agent 判断通过、返工、暂停或推进 |

这里的关键点是：

> PDCA 不是停留在管理学概念里，而是被 Agent 工程化了。

也就是说，每个阶段都不是“做完就算了”，而是必须经过：

```text
阶段任务生成
↓
执行
↓
检查
↓
返工或通过
↓
状态记录
↓
进入下一阶段
```

如果通过，就进入下一个 P。
如果不通过，就返回当前 P 返工。
如果风险较高，就暂停并请求人工确认。

---

## 七、它不是一层 Loop，而是三层 Loop

很多人一开始理解 Agent Loop，会以为只有执行层的 PDCA 是循环。

但实际上，完整的 Agent Loop 至少有三层循环。

```text
Objective Loop
  └── KR Loop
        └── P Loop（PDCA）
```

也就是说：

| 层级               | 循环对象       | 解决什么问题        |
| ---------------- | ---------- | ------------- |
| P 层 Loop         | 每一个阶段 P    | 这个小阶段有没有做好    |
| KR 层 Loop        | 每一个关键结果 KR | 这个关键结果有没有达成   |
| Objective 层 Loop | 整个目标       | 项目最终目标有没有真正完成 |

P 阶段完成后，不代表 KR 一定完成。
所有 KR 完成后，也不代表 Objective 一定达成。

所以必须有三层验证：

```text
P Validation：这个阶段是否完成？
KR Validation：这个关键结果是否达成？
Objective Validation：整个目标是否真正实现？
```

这就是分层推进、逐级验收。

---

## 八、第五层：状态层 State

为了让 Agent 能够持续推进，必须有状态记录。

状态层解决的是：

> 当前做到哪一步了？下一步应该做什么？

如果没有状态文件，Agent 很容易变成临时对话。
今天做了一半，明天再继续，它可能已经忘记：

* 当前 Objective 是什么；
* 当前做到哪个 KR；
* 当前在哪个 P 阶段；
* 哪些已经完成；
* 哪些返工过；
* 哪些风险还没有解决；
* 下一步应该做什么；
* 是否需要人工确认。

所以，每个 Agent Loop 项目都应该维护状态文件，比如：

```text
.agents/agent-state.md
.agents/mvps/mvp-01.md
```

状态文件不是聊天记录，而是工作状态记录。

它可以包含：

```md
# Agent State

## 当前状态

- Objective：
- 当前 KR：
- 当前 Plan：
- 当前 P 阶段：
- 当前阻塞点：
- 是否需要人工确认：

## 已完成内容

-

## 最近一次执行结果

-

## 最近一次审核结果

-

## 主 Agent 判断

-

## 下一步动作

-
```

状态层让 Agent Loop 可以跨轮恢复、持续推进。

---

## 九、第六层：审计层 Audit Trail

除了状态层，我认为还必须加一个更重要的层级：

> **审计层，也可以叫过程留痕层。**

状态层解决的是：

> 现在做到哪了？

审计层解决的是：

> 之前每一步是谁做的、做了什么、为什么这么做、检查发现了什么、为什么返工、最后为什么通过？

这非常关键。

因为 Agent Loop 不能是一个黑箱。
尤其当它从写代码推广到日常工作、企业工作、制度编写、方案生成、项目推进时，每一步都应该有记录。

每一次执行、检查、返工和决策，都应该在后台形成一份 Markdown 记录。

这份记录让整个过程：

* 可回溯；
* 可复盘；
* 可审计；
* 可追责；
* 可优化。

这也是 Agent Loop 能真正用于复杂工作的关键。

---

## 十、Markdown 审计记录应该记录什么？

每一轮 Loop 都可以形成一份 Markdown 审计记录。

它可以包含：

```md
# Agent Loop 审计记录

## 基本信息

- 时间：
- Objective：
- 当前 KR：
- 当前 P 阶段：
- 主 Agent：
- 执行 Agent：
- 审核 Agent：

## 本轮阶段提示词

-

## 执行记录

- 执行了什么：
- 修改/生成了什么：
- 输出文件：
- 自测方式：
- 自测结果：

## 审核记录

- 是否满足本阶段目标：
- 是否满足当前 KR：
- 阻断问题：
- 非阻断建议：
- 风险点：

## 返工记录

- 是否返工：
- 返工原因：
- 返工次数：
- 本轮返工结果：

## 主 Agent 判断

- 结论：通过 / 返工 / 暂停 / 人工确认
- 判断理由：
- 下一步：

## 人工确认

- 是否需要人工确认：
- 确认事项：
- 确认结果：
```

这就意味着：

> Agent 做过什么，不是只存在于聊天窗口里，而是沉淀为一份可检查的工作记录。

这个能力对写代码有用，对日常工作同样有用。

---

## 十一、为什么审计层很重要？

因为很多复杂工作，不只是要结果，还要过程。

比如：

| 场景    | 为什么需要审计记录              |
| ----- | ---------------------- |
| 写代码   | 要知道改了哪些文件、为什么改、测试是否通过  |
| 写制度   | 要知道依据是什么、风险点是什么、谁审核过   |
| 做方案   | 要知道每一版为什么调整，哪些建议被采纳    |
| 做 PRD | 要知道需求如何拆解，哪些功能被推迟      |
| 做课程   | 要知道大纲、讲稿、PPT 是如何一步步形成的 |
| 做合规材料 | 要知道检查依据、风险判断和最终结论      |

所以，审计层让 Agent Loop 不只是“自动干活”，而是“可信干活”。

它解决了一个核心顾虑：

> AI 自动推进，会不会乱来？

答案是：

> 它可以自动推进，但每一步都要留下记录。
> 做了什么、为什么做、谁检查了、哪里返工了、最后为什么通过，全部有迹可查。

---

## 十二、提示词不再主要由人写，而是由 Agent 自动生成

在传统 AI 使用方式里，人要不断写提示词。

但在 Agent Loop 里，每一轮推进所需要的提示词，不再主要由人临时手写，而是由主 Agent 或提示词编排 Agent 自动生成。

它会基于：

* Objective；
* 当前 KR；
* 当前 Plan；
* 当前 P 阶段；
* PRD；
* 状态文件；
* 审计记录；
* 历史返工记录；
* 审核 Agent 的反馈；
* 当前风险；
* 下一步动作。

然后生成一份阶段任务单。

这个提示词往往比人临时写的更严谨。

不是因为 AI 天然比人更聪明，而是因为它有几个优势：

| 优势    | 说明                    |
| ----- | --------------------- |
| 上下文完整 | 可以读取目标、计划、状态和历史记录     |
| 格式稳定  | 每次都按固定结构生成任务单         |
| 边界清楚  | 会写明做什么、不做什么、允许范围、禁止范围 |
| 验收明确  | 会把 KR 和 P 阶段目标转成可检查标准 |
| 风险继承  | 上一轮发现的问题会进入下一轮提示词     |
| 可追踪   | 每轮提示词都会写入审计记录         |

所以，Agent Loop 的一个重要变化是：

> 人不再每一步写提示词，而是由 Agent 基于目标和状态自动生成更严谨的阶段提示词。

---

## 十三、人类在这个系统里做什么？

这套系统不是完全不需要人，而是让人从“操作员”变成“关键节点审批人”。

传统方式里，人经常要做：

* 每一步都写提示词；
* 每一步都解释背景；
* 每一步都判断下一步；
* 每一步都检查结果；
* 每一步都提醒 AI 不要跑偏。

但在 Agent Loop 里，人类主要做三件事：

| 人类职责 | 说明                  |
| ---- | ------------------- |
| 设定目标 | 明确 Objective 和最终期望  |
| 设定边界 | 明确哪些事情不能自动做         |
| 关键确认 | 在高风险节点、方向变化、最终验收时介入 |

所以，这套系统追求的是：

> 低风险阶段自动推进，高风险节点人工确认。

比如这些事情必须人工确认：

* 上线发布；
* 删除数据；
* 数据库迁移；
* 修改权限；
* 引入高风险依赖；
* 发送外部消息；
* 涉及真实付款；
* 处理真实用户隐私数据；
* 改变产品方向或商业模式。

人不是消失了，而是站到了更高层：

> 人负责方向，Agent 负责推进。
> 人负责关键确认，Agent 负责执行闭环。

---

## 十四、它可以用于写代码，也可以用于日常工作

这套机制最容易在写代码中被理解。

因为代码开发天然有：

* 需求；
* 拆解；
* 开发；
* 测试；
* 审核；
* 返工；
* 记录。

但它不只适用于写代码。

任何复杂工作，只要有目标、有阶段、有验收、有风险，都可以使用 Agent Loop。

比如写一份工作方案：

```text
Objective：完成一份可上会讨论的工作方案

KR1：形成完整方案框架
KR2：完成正文初稿
KR3：补充风险分析和保障措施
KR4：形成领导汇报版材料
```

然后每个 KR 再拆成 Plan，每个 P 阶段进入 PDCA。

整个过程中，每一次执行、审核、返工、确认都会形成 Markdown 审计记录。

最终得到的不只是一份方案，还有一条完整的过程链：

* 第一版是怎么来的；
* 审核发现了什么；
* 哪些地方被返工；
* 哪些建议被采纳；
* 哪些风险被保留；
* 为什么最终可以提交。

这就非常适合真实工作场景。

---

## 十五、这套系统适合什么任务？

它不适合所有任务。

如果只是：

* 改一句话；
* 写一个标题；
* 翻译一段文字；
* 查一个简单概念；

没有必要上这套流程。

但只要任务具备以下特征，就适合使用 Agent Loop：

| 任务特点    | 示例                  |
| ------- | ------------------- |
| 目标复杂    | 开发应用、搭建知识库、做课程方案    |
| 需要分阶段推进 | 调研、设计、初稿、审核、修改、定稿   |
| 需要质量检查  | 代码审核、合规审核、逻辑审核、格式审核 |
| 需要跨轮恢复  | 今天做一半，明天继续          |
| 需要过程留痕  | 制度、方案、项目推进、合规材料     |
| 不能一次生成完 | PRD、PPT、系统开发、投标材料   |

简单来说：

> 小事直接问 AI，复杂任务交给 Agent Loop。

## 十六、完整链路总结

这套系统的完整链路可以表达为：

```text
人类设定 Objective 和边界
↓
主 Agent 拆解 KR
↓
为每个 KR 制定 Plan
↓
主 Agent 自动生成阶段提示词
↓
执行 Agent 完成任务
↓
审核 Agent 独立检查
↓
主 Agent 判断通过 / 返工 / 暂停
↓
状态文件记录当前进度
↓
Markdown 审计记录沉淀过程
↓
通过则进入下一个 P
↓
当前 KR 完成后进行 KR Validation
↓
KR 通过后进入下一个 KR
↓
全部 KR 完成后进行 Objective Validation
↓
目标验证通过，项目完成
```

这就是一个目标驱动、自动推进、带自检、带留痕的 Agent 工作系统。

## 十七、一句话总结

普通 AI 使用，是：

> 我问一句，AI 答一句。

Agent Loop 是：

> 我给一个目标，AI 自动拆解、执行、检查、返工、记录，并持续推进到目标完成。

更完整地说：

> **OKR-PDCA Agent Loop，是一种目标驱动的自推进 Agent 工作系统。它用 Objective 明确目标，用 KR 定义关键结果，用 Plan 拆解执行路径，用 PDCA 完成阶段闭环，用状态文件支持持续推进，用 Markdown 审计记录保证全过程可回溯、可复盘、可审计。**

---

## 十八、适合演讲收尾的一段话

未来真正有价值的 AI 使用方式，不是人类不断写提示词，让 AI 一步一步回答。

而是人类设定目标和边界，AI 根据目标自动拆解任务、生成提示词、执行任务、检查结果、持续返工，并在每一步留下可追踪的过程记录。

这时，AI 不再只是一个聊天工具。
它会变成一个围绕目标持续推进、能够自检、能够返工、能够留痕的工作系统。

一句话：

> **人负责方向，Agent 负责推进；人负责关键确认，Agent 负责执行闭环；过程全程留痕，结果才能可信。**
